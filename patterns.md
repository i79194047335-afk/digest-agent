## Структура файлов проекта

### CLAUDE.md — контракт агента

Главный файл который агент читает при старте. Здесь общие правила: куда писать файлы, на каком языке, какова точка входа. Агент читает его один раз и следует ему всю сессию.

```markdown
# Digest Agent — Claude               (название агента)

## STRICT RULES                       (жёсткие правила — агент не может их нарушить)
- All results go to ../workspace/claude/        (куда сохранять все файлы)
- Write in Russian                              (язык вывода)
- Before each action write one line: ...        (логировать каждый шаг вслух)

## Entry point                        (точка входа — с чего начинать каждую сессию)
Always start with ORCHESTRATOR.md — it decides what to do next.
```

### Файлы инструкций — что и как делать

Это пошаговые инструкции для конкретной задачи. Агент получает сюда управление от оркестратора и выполняет шаги строго по файлу.

```markdown
# step1_plan.md                       (название шага — агент понимает что это первый шаг)

## Steps                              (список шагов — выполняются строго по порядку)
1. Read the topic from the user request    (взять тему из того что написал пользователь)
2. Create a plan with 5 sections           (сформировать структуру будущего текста)
3. Save to ../workspace/claude/plan.md     (точное имя файла — агент не придумывает своё)
```

### README.md — архивная документация блока

Создаётся агентом (`doc_agent.md`) после завершения каждого блока. Описывает что было сделано, какие файлы созданы, какие выводы получены. Хранится в архиве блока в git.

```markdown
# Block 04 — Prompt Chaining          (номер блока и название паттерна)

## What was done                      (краткое описание что делали)
Two-step pipeline: plan → write.      (план из двух шагов)

## Files                              (все файлы которые участвовали в блоке)
- instruction_files/claude/step1_plan.md    (файл инструкций для шага 1)
- results/claude/digest.md                 (итоговый результат работы агента)

## Key insight                        (главный вывод — что узнали из этого блока)
Agent reads CLAUDE.md once at session start.    (агент читает контракт только один раз)
Changes require full restart.                   (правки вступают в силу только после рестарта)
```

## Блок 4 — Prompt Chaining (цепочка)

Это всегда линейная последовательность шагов. Шаг 1 вызывает шаг 2, шаг 2 вызывает шаг 3. Выход одного шага становится входом следующего.

Шаги прописаны заранее и всегда одинаковые — агент не принимает никаких решений, он просто идёт по цепочке.

В нашем проекте: `step1_plan.md` делает план → план сохраняется в `plan.md` → `step2_write.md` читает план и пишет текст → результат в `digest.md`.

**Когда это нужно:** задача одна и сложная. Разбиваешь её на части чтобы не перегружать модель одним огромным промптом. Каждую часть можно проверить отдельно.

## Блок 5 — Routing (маршрутизация)

Изначально прописываешь инструкции для каждого возможного типа запроса. Когда приходит запрос — система смотрит на ключевое слово и выбирает нужный путь из заранее готовых. Ничего нового не придумывает — только выбирает.

В нашем проекте: пишешь `quick` — идёт по одному пути, пишешь `deep` — по другому. Оркестратор (`ORCHESTRATOR.md`) читает запрос и направляет в `quick_digest.md` или `deep_digest.md`.

**Routing — это по сути система из нескольких Chaining.** Каждый маршрут внутри — это своя цепочка шагов. Routing только решает в какую цепочку войти.

**Когда это нужно:** когда запросы бывают принципиально разные и каждый тип требует своего подхода. Вместо одного универсального агента — несколько специализированных pipeline.
